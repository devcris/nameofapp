Production-Ready Web App

6.5: Payment Forms with Stripe

Learning Goals

  • Understand how Stripe handles credit card payments online
  • Register an account on Stripe
  • Configure your application to use the Stripe API
  • Implement a “Pay with Stripe” button on your site

 Estimated Read Time: 15 Minutes.

Introduction

Hello again! Where it was once a rarity, and prone to bugs and hackers, online transactions have become the norm. They are highly regulated and many safety standards have been put into place to protect our identities.

Whether or not you regularly charge your bank card online, you surely have a sense of how highly sensitive that data is, and why it must be treated with care and consideration. While there are various options for handling this data, we are big fans of Stripe.

What Is Stripe?

Stripe is a payment processing service. You can use their system to handle credit card payments on your site. There are a variety of payment processing APIs available (PayPal and Authorize.net are a couple of examples), but Stripe has a couple key benefits over other popular ones:

  1. Stripe transfers payments directly to your bank account after a 7-day waiting period.
  2. Stripe is an API first and foremost, and it's designed to be as simple as possible to integrate.

The second point is important because it makes life for developers very easy. In particular, their API processes payments in a secure fashion that avoids ever having to store the user's credit card data on your site—it gets transmitted directly to Stripe via an encrypted token on the client side. This is advantageous because it bypasses the database of your application and passes the burden of legal compliance from the website administrator onto Stripe (any website which handles credit card data must be in compliance with PCI security standards).

Read Stripe's Features Overview. This will give you an idea of the benefits of using them, particularly for ease of integration and security.

Setting Up Stripe

Before we dive into implementing our payments system, we need an account with Stripe. Because we're dealing with financial information and payments, you need a little bit more than a username and password to sign up.

Head over to Stripe's registration page and go through the signup process. If you have trouble with any steps, ask your Mentor for assistance.

"Pay Now" Button on Your Product Page

Now that you're set up with Stripe, we're going to configure our Rails app to connect with Stripe, then add a "Pay Now" button to product pages that will allow us to collect payments.

To get started, go to your Stripe dashboard and find your API keys (towards the bottom of the left sidebar). You'll see two sets of keys. One for "test" mode and the other for "live" mode. As you might expect, the test keys can be used in development mode to simulate payments without any money changing hands.

To get access to the keys in our app, we have a number of options. The easiest thing to do would be to hard-code them wherever they're needed, but this is a terrible idea for a number of reasons. Specifically, we don't want our Stripe secret to be shared with other people and by adding it to the code it'll eventually end up on GitHub and be available to anyone who has access to that. We also want to be able to switch between using the test keys and the production keys depending on which environment the app is deployed to. Instead of hard coding, we'll be using environment variables on Heroku to store your production keys.

Initializing Stripe

First, add the Stripe gem to your Gemfile: gem 'stripe' and then run bundle install from the terminal.

Next, create a new initializer to configure the Stripe gem to use your API keys. The initializer will use your test keys in development and your live keys in production. Put the following code into a new file at config/initializers/stripe.rb:

if Rails.env.production?
  Rails.configuration.stripe = {
    publishable_key: ENV['STRIPE_PUBLISHABLE_KEY'],
    secret_key: ENV['STRIPE_SECRET_KEY']
  }
else
  Rails.configuration.stripe = {
    publishable_key: 'pk_test_zkT...',
    secret_key: 'sk_test_I0e...'
  }
end

Stripe.api_key = Rails.configuration.stripe[:secret_key]

Copy your test key and secret into the second part of the conditional. You can now reference your Stripe credentials as Rails.configuration.stripe[:publishable_key] anywhere in your app.

Environment Variables

In the code above, you see a conditional that checks whether the Rails app is running in the production environment. If it does NOT run on production, it'll use the test key and secret we added to the code. If it runs on production, the first part of the code will run, and instead of the actual publishable key and the secret key, you see two so-called environment variables:
ENV['STRIPE_PUBLISHABLE_KEY'] and ENV['STRIPE_SECRET_KEY'].

Environment variables are global variables defined in an environment separate from the actual code. This way, you can keep your source code open without showing secret keys to everyone. In Heroku, setting up environment variables is very easy.

First, log into your Heroku Dashboard and select your app. Next, click on "Settings" on the top right. Then, click on "Reveal Config Vars".

Now you can define environment variables. On the left (as the key), type in "STRIPE_PUBLISHABLE_KEY" and on the right (as the value) the key from Stripe. Do the same thing for the STRIPE\_SECRET\_KEY variable.

You can use your test keys as values as long as you're developing your app or want to present it on your portfolio. If your app is ready to be published, you can change the keys to the actual Stripe keys for real payments.

Frontend Implementation

Take a look at the Stripe tutorial on Integrating Checkout. This illustrates the quickest and easiest implementation where you can add a simple JavaScript snippet to your page, displaying a "Pay" button. When clicked, that button opens up a Stripe widget which has a credit card form and pay button.

To begin, create a partial called _stripe_checkout_button.html.erb in your app/views/shared folder. Paste the following code based on the example from the Stripe docs into it:

<script
  src="https://checkout.stripe.com/checkout.js" class="stripe-button"
  data-key="<%= Rails.configuration.stripe[:publishable_key] %>"
  data-image="/square-image.png"
  data-name="Demo Site"
  data-description="2 widgets ($20.00)"
  data-amount="2000">
</script>

Note: we're loading our Stripe publishable key into the data-key attribute in the script. This is how we identify ourselves to Stripe.

There are several other variables here that are currently hard-coded. We'll need to change these, as this partial could be embedded into all kinds of different product pages. Using your knowledge of partials and views, replace the values of data-name, data-description, and data-amount with product-specific variables.

On your product Show page, add a new form tag and render the Stripe button partial inside it. It'll look something like this:

<%= form_with(url: '/payments/create') do |form| %>
  <%= render partial: "shared/stripe_checkout_button" %>
<% end %>

If you run your app and check the product pages, you'll see that there's now a blue 'Pay with Card' button on each of them.

Troubleshooting Datatypes for Money

Make sure the data type of the column price is numeric. Best practices dictate that currency is saved in integer form (and that you use naming like "price_in_cents" to make it clear what the field contains). Below are some notes that speak to the pros and cons of the different data types with regard to currency.

Integer

Stripe expects the price you give it to be an integer in cents. Thus, saving the price as an integer and in cents in the first place makes this step easy. Keep in mind, however, that for the views, whenever you display the price, you'll have to convert the displayed amount to whatever currency you're using. (Usually, this means dividing the price integer by 100 and converting it into a decimal that only shows two digits after the dot.)

Decimal

Keeping your prices as decimals in the database might make a lot of calculations and displaying the price on your website easier, as it will already be in the correct format. You still need to make sure, though, that the decimal has only two digits after the dot. In your Stripe form, you'll have to convert the price back to cents so that the numbers make a clean integer.

Float

Never use floats for money or price! It could cause serious problems in calculations. To read more about reasons not to use floats, check out what software developer Patrick Bacon has to say in Floating Point Numbers and Currency Rounding Errors.

Go ahead and make sure your price is the correct data type, then adjust the payment forms accordingly.
